## 3. Critical Issues Identified

### Issue 1: Speedup Does Not Scale with Sparsity

| Views | Sparsity (k=3) | Expected Speedup | Actual Speedup |
|-------|----------------|------------------|----------------|
| 8 | 57% | ~2x | 1.42x |
| 16 | 80% | ~5x | 1.11x |
| 32 | 90% | ~10x | 1.01x |
| 64 | 95% | ~20x | 1.00x |

**Root Cause Analysis:**
1. The sparse attention mask may not be properly integrated with PyTorch's attention computation
2. Memory access patterns may not benefit from sparsity on GPU
3. The mask overhead may cancel out sparsity benefits

---

### Issue 2: GT Evaluation is Broken

All results show:
- `gt_mae`: ~18,000-19,000 (extremely high)
- `gt_abs_rel`: 0.9999 (99.99% relative error)

This indicates:
1. **Scale mismatch** between predicted depth and GT depth
2. Predicted depth may be in different units (e.g., disparity vs. meters)
3. No depth alignment/normalization is being applied

**Fix Required:** Implement median scaling alignment before computing metrics:
```python
scale = np.median(gt_depth) / np.median(pred_depth)
pred_aligned = pred_depth * scale
```

---

### Issue 3: Identical Quality Across All k Values

All sparse configurations produce **identical** gt_mae/gt_rmse values as dense:
- Dense: gt_mae = 18,799.74
- k=3: gt_mae = 18,799.74
- k=5: gt_mae = 18,799.74
- k=10: gt_mae = 18,799.74

This confirms that sparse attention produces **identical outputs** to dense attention, which is good for quality preservation but raises questions about whether the sparse path is being used.
