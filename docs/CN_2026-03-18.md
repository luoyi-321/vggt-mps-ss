## 3. 发现的关键问题

### 问题1：加速比不随稀疏度增加

| 视图数 | 稀疏度 (k=3) | 预期加速比 | 实际加速比 |
|--------|--------------|------------|------------|
| 8 | 57% | ~2x | 1.42x |
| 16 | 80% | ~5x | 1.11x |
| 32 | 90% | ~10x | 1.01x |
| 64 | 95% | ~20x | 1.00x |

**根本原因分析:**
1. 稀疏注意力掩码可能未正确集成到PyTorch的注意力计算中
2. GPU上的内存访问模式可能无法从稀疏性中受益
3. 掩码开销可能抵消了稀疏性带来的好处

---

### 问题2：GT评估结果异常

所有结果显示：
- `gt_mae`: ~18,000-19,000（极高）
- `gt_abs_rel`: 0.9999（99.99%相对误差）

这表明：
1. 预测深度和GT深度之间存在**尺度不匹配**
2. 预测深度可能使用不同的单位（如视差 vs 米）
3. 没有应用深度对齐/归一化

**需要修复:** 在计算指标前实现中值缩放对齐：
```python
scale = np.median(gt_depth) / np.median(pred_depth)
pred_aligned = pred_depth * scale
```

---

### 问题3：所有k值的质量相同

所有稀疏配置产生与稠密**完全相同的**gt_mae/gt_rmse值：
- 稠密: gt_mae = 18,799.74
- k=3: gt_mae = 18,799.74
- k=5: gt_mae = 18,799.74
- k=10: gt_mae = 18,799.74

这确认了稀疏注意力产生与稠密注意力**完全相同的输出**，这对质量保持是好事，但也引发了稀疏路径是否真正被使用的疑问。

## 6. 后续步骤

### 优先级1：修复稀疏注意力加速
稀疏掩码计算正确但未转化为加速。需要调查：
- PyTorch是否使用掩码进行计算节省？
- FlashAttention是否与稀疏掩码兼容？
- 考虑实现自定义稀疏注意力内核

### 优先级2：修复GT深度评估
实现正确的深度对齐：
```bash
python scripts/evaluate_with_gt.py \
    --align-scale median \
    --data-dir data/co3d_benchmark
```

## 7. 结论

实验成功展示了：
- 在CO3D数据上的真实VGGT-1B推理
- 正确的稀疏掩码计算（高达95%）
- 稀疏注意力的质量保持

然而，**加速比显著低于预期**。稀疏注意力实现需要优化，以将理论稀疏度转化为实际加速。

**论文建议:** 在加速问题解决之前，重点关注内存节省和可扩展性论述，而非加速声明。
