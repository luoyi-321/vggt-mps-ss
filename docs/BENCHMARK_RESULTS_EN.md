## Overview

The VGGT-MPS benchmarking system evaluates the performance and quality trade-offs of sparse attention mechanisms compared to dense (full) attention. The key insight is:

- **Dense Attention**: O(n²) memory complexity - every image attends to every other image
- **Sparse Attention**: O(n·k) memory complexity - each image only attends to k nearest neighbors

This enables processing **10-100x more images** on the same hardware.

## Benchmark Modes

### 1. `basic` - Basic Performance Benchmark

**Command:**
```bash
vggt benchmark --mode basic --images 30 --compare dense,sparse
```

**What it measures:**
- Processing time for dense vs sparse attention
- Memory usage (on CUDA devices)
- Frames per second (FPS)

**Output metrics:**
| Metric | Description |
|--------|-------------|
| Time | Total processing time in seconds |
| FPS | Images processed per second |
| Memory | GPU memory used (CUDA only) |
| Speedup | How much faster sparse is vs dense |

### 2. `consistency` - Quality Consistency Benchmark

**Command:**
```bash
vggt benchmark --mode consistency \
    --images 5,10,20,30 \
    --compare dense,sparse \
    --metrics depth_l1,pose_rotation,pose_translation,chamfer
```

**What it measures:**
Compares output quality between dense and sparse methods across different image counts.

**Output metrics:**
| Metric | Description | Lower = Better |
|--------|-------------|----------------|
| `depth_l1` | Mean absolute error in depth prediction | Yes |
| `pose_rotation` | Rotation error in camera poses | Yes |
| `pose_translation` | Translation error in camera poses | Yes |
| `chamfer` | Chamfer distance between point clouds | Yes |

**Note:** Metrics show "N/A" in simulated mode because poses and point clouds require the real VGGT model.

### 3. `ablation-k` - K-Nearest Parameter Ablation

**Command:**
```bash
vggt benchmark --mode ablation-k \
    --images 30 \
    --sparse-k 3,5,10,15,20,30
```

**What it measures:**
How the k-nearest parameter affects quality and efficiency.

**Key insights:**
- **Lower k** = More sparsity = Faster but potentially lower quality
- **Higher k** = Less sparsity = Slower but closer to dense quality
- Sweet spot is typically k=10-15 for good quality/speed trade-off

**Output table explanation:**
```
       k   Time(s)    Speedup   Sparsity   MemSavings     DepthL1
   dense      0.05      1.00x       0.0%         1.0x    0.000000  <- Baseline
       5      0.03      1.67x      83.3%         6.0x    0.012345  <- High sparsity
      10      0.04      1.25x      66.7%         3.0x    0.005678  <- Balanced
      20      0.04      1.10x      33.3%         1.5x    0.001234  <- Low sparsity
```

**How to interpret:**
- `Sparsity 83.3%` means 83.3% of attention connections are removed
- `MemSavings 6.0x` means using 6x less memory than dense
- `DepthL1 0.012345` means average depth error of 0.012345 meters

### 4. `ablation-tau` - Threshold Parameter Ablation

**Command:**
```bash
vggt benchmark --mode ablation-tau \
    --images 30 \
    --threshold 0.3,0.5,0.7,0.8,0.9
```

**What it measures:**
How the covisibility threshold (τ) affects which images are connected.

**Key insights:**
- **Lower τ (0.3)** = More connections kept = More memory but better quality
- **Higher τ (0.9)** = Fewer connections = Less memory but may miss important relationships
- Threshold controls the "similarity cutoff" for image pairs

**How threshold works:**
```
If similarity(image_i, image_j) >= τ:
    Keep connection (attend to each other)
Else:
    Remove connection (no attention)
```

### 5. `ablation-mask` - Mask Type Comparison

**Command:**
```bash
vggt benchmark --mode ablation-mask \
    --images 30 \
    --mask-types covisibility,random,sliding_window \
    --sparsity 0.56
```

**What it measures:**
Compares different strategies for creating sparse attention masks at the **same sparsity level**.

**Mask types:**

| Type | Description | Best For |
|------|-------------|----------|
| `covisibility` | Connect images that see similar scene content | Real-world sequences |
| `random` | Randomly select which images connect | Baseline comparison |
| `sliding_window` | Connect nearby frames in sequence | Video/temporal data |

**Why this matters:**
Even at the same sparsity (e.g., 56%), **which** connections you keep dramatically affects quality. Covisibility-based masks outperform random masks because they preserve semantically meaningful connections.

### 6. `compare-methods` - Full Method Comparison

**Command:**
```bash
vggt benchmark --mode compare-methods \
    --images 30 \
    --methods dense,sliding_window,random,covisibility \
    --sparsity 0.5,0.6,0.7
```

**What it measures:**
Comprehensive comparison of all methods across multiple sparsity levels.

**Output includes:**
1. Performance table for each method × sparsity combination
2. Quality ranking at each sparsity level
3. JSON export for further analysis

**Example ranking output:**
```
Method Ranking by Quality (lowest error at each sparsity level):
  50% sparsity: covisibility(0.0052) > sliding_window(0.0078) > random(0.0134)
  60% sparsity: covisibility(0.0061) > sliding_window(0.0089) > random(0.0156)
  70% sparsity: covisibility(0.0073) > sliding_window(0.0102) > random(0.0189)
```

**Interpretation:** Covisibility consistently produces the lowest error, followed by sliding_window, then random.

## Understanding Metrics

### Memory Metrics

| Metric | Formula | Description |
|--------|---------|-------------|
| Dense Memory | O(n²) | Quadratic growth with image count |
| Sparse Memory | O(n·k) | Linear growth with image count |
| Memory Savings | n/k | How many times less memory is used |

**Example:** With n=100 images and k=10:
- Dense: 100 × 100 = 10,000 attention entries
- Sparse: 100 × 10 = 1,000 attention entries
- Savings: 10x less memory

### Quality Metrics

| Metric | Range | Meaning |
|--------|-------|---------|
| Depth L1 | 0 to ∞ | Mean absolute depth error (meters). 0 = perfect |
| Sparsity | 0% to 100% | Percentage of removed connections. Higher = more sparse |
| Connectivity | 0% to 100% | Average connections per image. Higher = more connected |

### Performance Metrics

| Metric | Meaning |
|--------|---------|
| Speedup | Times faster than dense baseline |
| Time | Actual processing time in seconds |
| Theoretical | Based on complexity analysis |
| Measured | Based on actual benchmark run |

## Interpreting Results

### Good Results Look Like:

```
Method: covisibility, Sparsity: 60%
- DepthL1: 0.005 (very low error)
- Speedup: 2.5x (significant speedup)
- MemSavings: 2.5x (good memory reduction)
```

### Warning Signs:

```
Method: random, Sparsity: 80%
- DepthL1: 0.150 (high error - quality degraded)
- Speedup: 5x (fast, but at what cost?)
```

### Optimal Trade-off Zone:

For most applications, aim for:
- **Sparsity: 50-70%** - Good balance
- **k: 10-20** - Sufficient neighborhood
- **τ: 0.5-0.7** - Reasonable similarity threshold

## Recommendations

1. **For real-time applications:** Use k=5-10, accept ~70% sparsity
2. **For high-quality reconstruction:** Use k=15-20, accept ~50% sparsity
3. **For maximum images:** Use k=5, τ=0.8, achieve ~85% sparsity
4. **Always use covisibility** over random masks when possible

## Troubleshooting

### "N/A" in metrics
- The real VGGT model is not loaded
- Run `vggt download` to get the model
- Or install: `pip install vggt`

### Similar results for all methods
- In simulated mode, all methods produce similar synthetic outputs
- Real differences appear with the actual VGGT model

### Memory errors
- Reduce `--images` count
- Increase `--sparse-k` values (more aggressive sparsity)
- Use lower resolution images
